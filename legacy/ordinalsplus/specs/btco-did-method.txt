ยง Abstract
 The BTCO DID Method specification defines a decentralized identifier method that leverages Bitcoin (BTC) Ordinal Theory and Inscriptions to create, resolve, update and deactivate Decentralized Identifiers (DIDs). This method provides a robust foundation for digital identity management directly on the Bitcoin blockchain by using ordinal numbers to uniquely identify satoshis. The specification details the syntax, operations, and security considerations for implementing BTCO DIDs, enabling organizations and individuals to establish verifiable digital identities that are as decentralized, durable, and secure as Bitcoin itself.

 

ยง Overview
 The BTCO DID Method provides decentralized identifiers using Bitcoinโs blockchain and ordinal theory. It leverages Bitcoinโs security and network effects while maintaining full compliance with the W3C DID Core Specification.

 

ยง How It Works
 The method assigns DIDs to specific satoshis using their ordinal numbers. The DID Document and related data are stored as inscriptions on these satoshis, enabling:

  โข Creation: Register a DID by inscribing it on an unused satoshi
 โข Resolution: Retrieve the DID Document by reading the inscription
 โข Updates: Modify the DID Document through reinscriptions
 โข Deactivation: Mark a DID as inactive using a special reinscription
  

ยง Key Features
  โข Zero counterparty risk through direct Bitcoin blockchain usage
 โข Standard DID operations via ordinal inscriptions
 โข Flexible cryptographic key management
 โข Integration with BTCO DID Linked Resources
 โข Support for verifiable credentials and digital identity management
  

ยง Purpose
 The primary purpose of this method is to:

  โข Provide a secure and decentralized identity solution without relying on additional networks
 โข Enable verifiable credential issuance and verification
 โข Support digital identity management directly on Bitcoin
 โข Leverage Bitcoinโs immutability for identity persistence
  

ยง Specification
 This section defines the technical specifications for creating, resolving, updating and deactivating BTCO DIDs using Inscriptions.

 

ยง Method Syntax and Structure
 The syntax for the BTCO DID method follows the generic DID scheme with the satoshi ordinal number as the method specific identifier:

 did:btco:<sat-number>

 Where:

  โข did: The standard DID prefix
 โข btco: The BTCO method identifier
 โข <sat-number>: The ordinal number of the satoshi (must be a positive integer between 0 and 2099999997689999)
  Example:

 did:btco:1066296127976657

 NOTE Previous versions of this specification described support for name-based and decimal identifiers. Numerical ordinal format is now the only supported format for better interoperability with the Ordinals protocol.

  

ยง DID Document Format
 The DID Document MUST be a valid JSON-LD document following this structure:

 {
"@context": [
"https://www.w3.org/ns/did/v1",
"https://w3id.org/security/multikey/v1"
],
"id": "did:btco:1066296127976657",
"verificationMethod": [
{
"id": "did:btco:1066296127976657#0",
"type": "Multikey",
"controller": "did:btco:1066296127976657",
"publicKeyMultibase": "z6MkhaXgBZDvotDkL5257faiztiGiC2QtKLGpbnnEGta2doK"
}
],
"authentication": ["did:btco:1066296127976657#0"],
"assertionMethod": ["did:btco:1066296127976657#0"]
}

 This example shows a basic DID Document. Additional properties defined in the DID Core specification MAY be included, such as:

  โข service endpoints
 โข keyAgreement for encryption
 โข capabilityInvocation for authorization
 โข capabilityDelegation for delegation
 โข Additional verification methods and relationships
  Key components of this example:

  โข The required JSON-LD contexts that provide the vocabulary for DID and cryptographic features
 โข A primary verification method using the Multikey type for flexible key representation
 โข Basic verification relationships (authentication and assertionMethod) referencing the verification method
  NOTE Since the DID Document is stored as metadata in the inscription, it can be accessed as a DID Linked Resource using the /meta suffix. For example: did:btco:1066296127976657/0/meta will return the DID Document for the first inscription on the satoshi. See the DID Linked Resources specification for more details about accessing inscription content and metadata.

  NOTE The Multikey context (https://w3id.org/security/multikey/v1) provides a modern verification method type that supports multiple key formats through multibase encoding. This allows for flexible key representation while maintaining compatibility with different cryptographic algorithms.

  

ยง DID Operations
 

ยง Create (Register)
 To create a BTCO DID:

  โข Select a satoshi by its ordinal number (e.g., 1066296127976657)

MUST be a positive integer
MUST be less than or equal to 2099999997689999 (total number of satoshis)
MUST NOT already have a valid DID inscription
 โข Generate cryptographic key pair(s)
 โข Create a DID Document following the specified format
 โข Create an inscription with any valid content
 โข Include the DID Document as metadata in CBOR format
 โข Register the inscription on the Bitcoin network
  

ยง Read (Resolve)
 Resolution process:

  โข Parse the DID components

MUST be a valid BTCO DID format
MUST contain a valid satoshi number
 โข Locate the inscription on the specified satoshi using ord

Query the satoshiโs inscription history
Find the most recent inscription with metadata
 โข Decode and validate the CBOR metadata

MUST contain a valid DID Document
Document id MUST match the requested DID
 โข Return the resolved DID Document
  Possible errors:

  โข invalidDidSyntax: DID string does not match required format
 โข satoshiNotFound: Specified satoshi number does not exist
 โข noInscription: No inscription found on the satoshi
 โข invalidMetadata: Inscription metadata is not valid CBOR
 โข invalidDidDocument: Metadata does not contain valid DID Document
 โข deactivated: DID has been deactivated
  Success response format:

 {
"didDocument": {
// The resolved DID Document
},
"metadata": {
"created": "<timestamp>",
"updated": "<timestamp>",
"deactivated": false,
"nextVersionId": "<DID Linked Resource Identifier>",
"versionId": "<DID Linked Resource Identifier>",
"equivalentId": ["<alternative DID URLs>"]
}
}

 NOTE The versionId and nextVersionId fields use DID Linked Resource identifiers (e.g., did:btco:1066296127976657/0) to reference specific versions of the DID Document. This format allows direct access to historical DID Document versions through the DID Linked Resources specification.

  Error response format:

 {
"error": "<error code>",
"message": "<human readable message>",
"details": {
// Additional error context
}
}

 

ยง Update
 To update a BTCO DID:

  โข Create new DID Document
 โข Sign the update using an authorized verification method
 โข Create an inscription with any valid content
 โข Include updated DID Document as CBOR metadata
 โข Register as a reinscription on the target satoshi
  

ยง Deactivate
 To deactivate a BTCO DID:

  โข Create a minimal DID Document containing:
  {
"@context": [
"https://www.w3.org/ns/did/v1",
"https://w3id.org/did-resolution/v1"
],
"id": "did:btco:1066296127976657",
"deactivated": true
}

  โข Include this document as CBOR metadata in the inscription
 โข Register as a reinscription on the target satoshi
 โข The inscription content can be any valid data
  NOTE The deactivated state is defined in the DID Resolution specification and will be recognized by standard DID resolvers.

  

ยง Cryptographic Operations
 

ยง Control and Key Management
 The BTCO DID method ties DID control directly to Bitcoin UTXO ownership:

  โข The Bitcoin private key controlling the satoshiโs UTXO is the ultimate controller of the DID
 โข Only the current UTXO owner can update or deactivate the DID by creating new inscriptions
 โข Control of the DID transfers automatically when the satoshi moves to a new UTXO
 โข Loss of the Bitcoin private key controlling the UTXO means loss of DID control
  The DID Document itself can contain multiple verification methods as defined in the DID Core specification. These keys:

  โข Are independent from the Bitcoin key controlling the DID
 โข Can be updated by the UTXO owner through a new DID Document
 โข Can be rotated or revoked if compromised (as long as UTXO control is maintained)
  NOTE While Bitcoin UTXO recovery mechanisms are outside the scope of this specification, DID Document keys can be recovered through a DID Document update by the UTXO owner.

  

ยง Supported Algorithms
 The method supports multiple cryptographic suites through the Multikey verification method type:

  โข Ed25519 for digital signatures (z6Mk prefix)
 โข X25519 for key agreement (z6LS prefix)
 โข secp256k1 for Bitcoin compatibility (z6MW prefix)
  The Data Integrity proof suite MUST be used for all cryptographic operations. This provides:

  โข Standardized proof format
 โข Cryptographic agility
 โข Future-proof verification methods
  

ยง Key Management
 Implementations MUST:

  โข Generate secure random keys
 โข Support key rotation
 โข Implement secure key storage
 โข Enable key recovery mechanisms
 โข Use appropriate key types for different operations (signing vs encryption)
  

ยง Security Considerations
 Additional security requirements:

  โข Validate all cryptographic proofs
 โข Verify inscription ownership and order
 โข Implement proper key management
 โข Follow Bitcoin network security practices
 โข Monitor for blockchain reorganizations
 โข Validate DID Document format
  

ยง Privacy Considerations
 Enhanced privacy measures:

  โข Prohibit on-chain personal data
 โข Use separate Bitcoin addresses for DID operations
 โข Consider the permanent nature of blockchain data
 โข Implement appropriate data protection measures
 โข Support privacy-preserving cryptosuites
 โข Enable selective disclosure capabilities
  

ยง Inscription Content
 BTCO DID inscriptions can contain any valid data in their content. The DID operations (create, update, deactivate) are controlled through the inscription metadata, while the content remains unrestricted for application-specific uses.

 

ยง Inscription Requirements
 This DID method relies on the Ordinals protocol and the ord client for inscription handling:

 NOTE All inscription creation, validation, and ordering follows the Ordinals protocol as implemented by the reference ord client. This specification does not modify or extend the underlying inscription mechanisms.

  

ยง DID Document Storage
  โข The DID Document MUST be stored in the inscription metadata using CBOR encoding
 โข The inscription content MAY contain any valid data as defined by the Ordinals protocol
 โข Multiple inscriptions on the same satoshi are handled according to Ordinals protocol rules
 โข Size limitations and other technical constraints are inherited from the Ordinals protocol
  

ยง Inscription Order
 When resolving a DID:

  โข The most recent valid inscription containing DID metadata is authoritative
 โข Inscription order is determined by the Ordinals protocol
 โข Earlier inscriptions are considered historical versions of the DID Document
  For detailed information about inscription mechanics, refer to the Ordinals protocol documentation.

 

ยง Terminology
  Bitcoin (BTC) :  The original and most widely adopted blockchain network, known for its decentralized and immutable nature.  Decentralized Identifiers (DIDs) :  A type of globally unique identifier that enables verifiable, decentralized digital identity, as defined by the W3C DID Core specification.  DID Document :  A document containing information related to a DID, such as verification methods and service endpoints.  Inscription :  Arbitrary content added to BTC satoshis (sats) to create BTC-native digital artifacts.  Ordinal Number :  A unique integer between 0 and 2099999997689999 that identifies a specific satoshi based on the order in which it was mined.  Ordinal Theory :  A numbering scheme for satoshis that allows for individual satoshis to be tracked and inscribed with data.  Verification Method :  A mechanism that can be used to verify a proof, such as a digital signature or encryption.  Service Endpoint :  A network address at which services can be accessed using a DID.  Controller :  The entity that has the capability to make changes to a DID Document.
