Fix five correctness bugs across the VC, DID, CEL, and Bitcoin layers (three of them security-relevant), each with regression coverage:

- **Verifiable Credential issuer/holder binding.** The Data Integrity verify path (`Verifier.verifyCredential` / `verifyPresentation`) resolved the signing key solely from `proof.verificationMethod` and never checked that its controlling DID matched the credential `issuer` (or presentation `holder`). An attacker could sign a credential naming a trusted issuer with their own key and have it verify — full issuer impersonation. Verification now binds the proof's verification method to the issuer/holder before checking the signature, failing closed when the issuer/holder is absent (matching the signing side and the legacy verify path).
- **`did:webvh` resolution.** `DIDManager.resolveDID` called `didwebvh-ts` with no verifier, which the library requires to validate the log during resolution, so every valid `did:webvh` resolved to `null` (breaking credential verification, lifecycle, the document loader, and CEL key resolution for webvh issuers). It now passes the SDK's `Ed25519Verifier`.
- **CEL `did:btco` layer/DID detection.** `did:btco` migration events carry `sourceDid` (not `targetDid`, which is derived from the Bitcoin witness proof after inscription), but four detectors still keyed off `targetDid`. A `peer → webvh → btco` log was mis-detected as still at `webvh`, so `getCurrentState` returned the old webvh DID instead of `did:btco:<satoshi>`, the terminal-layer guard was bypassed (allowing a second inscription), and the CLI recorded the wrong `previousOwner` on transfers. All detectors now key off `sourceDid` and derive `did:btco:<satoshi>` from the witness proof.
- **Legacy credential expiry + 64-byte Ed25519 signing.** The legacy (non-cryptosuite) verify path validated the signature but skipped the validity window, so an expired legacy-signed credential verified as true; a shared validity check is now applied to both paths. Separately, signing with a 64-byte Ed25519 secret key (`seed || publicKey`) sliced off the seed half, producing signatures that never verified — it now uses the seed (first 32 bytes).
- **Fee-estimate bound + `did:btco` marker anchoring.** Fee rates returned by a fee oracle or ordinals provider were used without the upper bound already applied to caller-supplied rates; the bound now applies to estimator results too (an absurd estimate is skipped rather than used). The `did:btco` DID marker regex was start-anchored but not digit-bounded, so `did:btco:128` prefix-matched a `did:btco:1280` inscription and could mis-fire the deactivation path; a trailing digit boundary was added.
